# 封包掉包遊戲 · Packet Drop

**語言：** [English](README.md) · 繁體中文

> **太長一句話** — 一隊 6 位玩家「化身」成一條 TCP 連線。遊戲把一道謎題切成 6 個有編號的封包，亂序撒到玩家手機上，主持人還能「弄壞」其中幾個。要過關，全隊必須**把封包依序重組**，並且**一起從掉包中復原**——親手體會 TCP 是如何在不可靠的網路上做到可靠傳輸。

為 **SITCON Camp 2026** 打造。玩家只要用手機開網址即可，免安裝。最多 6 隊可同時進行。

---

## 🕹️ 核心流程（請先看這段）

> **太長一句話** — 每一回合：每位玩家收到 **6 個封包中的 1 個** → 按 **匯入** 把它丟進全隊共同筆記 → 全隊**拖曳排序**把亂掉的封包排回 `01→06` → 讀懂重組後的謎題 → 提交**一個團隊答案**。共做 3 回合（3 道謎題）。過程中隨時可能發生掉包事件（見下方）。

### 實際上在做什麼

一道謎題就是一則「訊息」，大到無法用單一封包送出。伺服器把它**切成 6 個片段**，每段標上序號（`封包 01/06` … `封包 06/06`），再**亂序地、一個座位發一段**。沒有任何一位玩家能看到完整謎題——他們必須把手上的片段匯集、重新排序，才能把訊息拼回來。

```
              伺服器  ── 本回合持有一道 6 片謎題
                 │
                 │  切成 6 個有編號的片段，
                 │  每個座位隨機分到 1 段
                 ▼
   ┌────────┬────────┬────────┬────────┬────────┬────────┐
  座位 1   座位 2   座位 3   座位 4   座位 5   座位 6
  封包03   封包01   封包05   封包02   封包06   封包04      ← 每支手機只看得到自己那段
   │        │        │        │        │        │
   └────────┴──── 每位玩家按下「匯入」 ──────┴────────┘
                 │
                 ▼
        共同筆記  ── 片段「亂序」抵達（匯入順序 ≠ 序號順序）
        03 · 01 · 05 · 02 · 06 · 04
                 │
                 │  全隊依序號「拖曳排序」
                 ▼
        01 · 02 · 03 · 04 · 05 · 06   ── 訊息可讀了：開場 → 線索 → 問題
                 │
                 ▼
        全隊一起「解謎」  →  提交「一個」團隊答案  →  下一回合
```

### 一步步拆解（單一回合）

| # | 步驟 | 由誰 | 畫面上發生什麼 |
|---|------|------|----------------|
| 0 | **派發** | 伺服器 | 回合開始時，伺服器發給每個座位一個片段。座位↔片段的對應是隨機但**穩定**的（用種子洗牌），回合中不會重洗。 |
| 1 | **接收** | 每位玩家 | 你手機上的 **「我的封包」** 卡片顯示你那一段，例如 `【封包 03/06】線索二：袋子 B 是綠色`。你看不到別人的。 |
| 2 | **匯入** | 每位玩家 | 按 **匯入至共同筆記**。你的片段被寫進全隊共同筆記——此時全隊都看得到它。 |
| 3 | **排序** | 全隊 | 片段依「匯入」順序出現在共同筆記，也就是亂的。全隊用 `封包 NN/06` 標籤把它們**拖曳排序**成 `01→06`。筆記每 2 秒同步，大家看到的順序一致。 |
| 4 | **解謎** | 全隊 | 排好後，片段就連成一則完整訊息（開場 → 4 條線索 → 問題）。全隊一起解這道邏輯題，並為全隊提交**一個**答案。 |
| 5 | **進關** | 主持人 | 主持人按 **下一關** → 伺服器載入下一道謎題，流程重複。第 3 關後，全隊看到完成畫面。 |

### 實例 — 第一關「神秘顏色」

謎題：A/B/C/D 四個袋子各裝一顆不同顏色的球（紅／藍／黃／綠）。6 個片段是：

| 序號 | 片段內容 |
|------|----------|
| `封包 01/06` | 開場：4 個袋子、4 種顏色，每袋各一 |
| `封包 02/06` | 線索一：袋子 A **不是**紅、**不是**藍 |
| `封包 03/06` | 線索二：袋子 B **是**綠 |
| `封包 04/06` | 線索三：袋子 C **不是**黃 |
| `封包 05/06` | 線索四：袋子 D **不是**綠、**不是**黃 |
| `封包 06/06` | 問題 + 作答格式 |

每位玩家隨機拿到其中一段。唯有全隊匯入 6 段並排成 `01→06`，線索鏈才有意義——接著就能推出 **A=黃、B=綠、C=紅、D=藍** 並提交。

## ⚠️ 掉包事件（高潮機制）

> **太長一句話** — 主持人可在回合中對某隊「掉包」。隨機幾個片段會變成亂碼、謎題變得不可讀。一條紅色橫幅會對**全隊**出現，而且**每一位玩家都必須按 ACK**，資料才會重傳、亂碼才會消失。只要有一個人不按，全隊就卡住。

依序發生的事：

1. **觸發。** 主持人對某隊按 **掉包**。伺服器隨機把 **50–80% 的座位**標記為「受影響」。
2. **出現亂碼。** 下一次 2 秒同步時，受影響的片段——無論是「我的封包」卡片*或*共同筆記裡——都會變成 `█▓▒░乱?#` 之類的雜訊。線索讀不了，謎題也就解不了。
3. **全員 ACK。** 一條紅色橫幅對**全隊**彈出（不只受影響的人），即時顯示進度如 `3/6 已確認`。這是**全體一致**的——每位已加入的玩家都得按 **ACK**。
4. **重傳。** 當**最後**一位玩家按下 ACK，伺服器將事件標記為已解決。下一次同步時亂碼清除、乾淨的片段回來。

這是遊戲的情感核心：全隊會體會到**連線會一直卡住，直到所有人都確認**，而且復原是**全隊**的責任，不只是被影響的人。

## 🎯 玩家學到什麼

> **太長一句話** — 四個 TCP 概念，用「體會」而非「背誦」：資料會被**分片**、封包會**亂序**抵達、封包可能**遺失／損毀**、復原需要**確認（ACK）**。藏在底下的一課：*可靠是用協調成本換來的。*

| TCP 概念 | 對應的遊戲機制 |
|----------|----------------|
| **分片（Fragmentation）** | 謎題太大塞不進一個封包，於是切成 6 段。沒人能單獨解出來。 |
| **亂序抵達** | 片段隨機落到各座位；共同筆記一開始是亂的，必須**依序號重排**——正是 TCP 接收端的工作。 |
| **掉包與損毀** | 主持人掉包；受影響的片段變成讀不懂的亂碼。 |
| **ACK 與重傳** | 復原需要**每一位**玩家 ACK，資料才重送——把 TCP 的「沒有確認就沒有進展」變成全體一致又看得到摸得到。 |

**為什麼這個比喻成立：**

- **序號** = `封包 NN/06` 標籤。能排序正是**因為**每段都有編號——就像 TCP 的位元組序號。
- **隨機抵達** = 網路不保證順序；把順序還原是接收端的責任。
- **共同筆記** = **接收緩衝區**，片段在此匯集，之後才依序交付給「應用層」（全隊的大腦）。
- **全體一致 ACK** = 刻意嚴格化的確認版本，好讓協調成本被「感受」到。真實 ACK 是逐段的，但原則——沒有 ACK 就沒有進展——完全相同。
- **可重現的損毀** = 損毀由固定種子計算而來，因此掉了的封包會**穩定地**保持壞掉（不閃爍），直到重傳。

## 🚀 如何執行

> **太長一句話** — `pnpm install` → `npx prisma migrate dev` → `pnpm dev`，然後玩家開 `/`、主持人開 `/admin`。

```bash
pnpm install
npx prisma migrate dev --name init   # 建立 SQLite 資料庫
pnpm dev                             # http://localhost:3000
```

| 路由 | 對象 | 用途 |
|------|------|------|
| `/` | 玩家 | 輸入組別 → 自動分配座位 → 候場大廳 → 開始 |
| `/game/[teamNumber]?slot=N` | 玩家 | 遊戲畫面：你的封包、共同筆記、作答框、ACK 橫幅 |
| `/admin` | 主持人 | 開始遊戲、觸發掉包、推進關卡、重置單隊或全部 |

### 主持人操作手冊
1. 把 `/admin` 投影到螢幕上。
2. 各桌挑一個組別號碼；玩家用手機加入（座位自動分配）。
3. 人到齊後對該隊按 **開始**。
4. 讓他們完成第一關的 **匯入 → 排序 → 解謎**。
5. 在回合中**觸發一次掉包**來教 ACK 機制——看著他們意識到*所有人*都得按 ACK。
6. 按 **下一關** 推進關卡。場次之間用 **重置**（或 **全部重置**）。

## 🛠️ 給開發者

> **太長一句話** — Next.js 14 + Prisma 5 + SQLite。狀態用每 2 秒輪詢同步（沒有 WebSocket）。片段分派與損毀都是可重現的（種子計算），所以「誰拿到哪一段」完全不需要存。完整架構見 [`CLAUDE.md`](CLAUDE.md)。

- **沒有 WebSocket** — 用戶端每 2 秒輪詢，並用「資料沒變就不更新」的守衛，因此只有資料真的變動時才重新渲染（避免拖曳排序每 2 秒重新量測 DOM）。
- **無狀態的片段分派** — 哪個座位拿哪一段，是由 `(teamNumber, round, slot)` 的種子洗牌推導出來的，不需儲存即可重現。
- 架構、檔案地圖、API 清單與資料庫結構見 [`CLAUDE.md`](CLAUDE.md)。
